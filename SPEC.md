# GSD Context Prune (上下文修剪插件) 规范

## 概述
`gsd-context-prune` 是一款为 Pi coding-agent 开发的插件，旨在缓解大模型在长时间运行的代理会话中面临的“上下文窗口膨胀”和“提示词缓存 (Prompt Cache) 命中率低下”两大问题。

它会监听助手 (Assistant) 的执行回合，捕获那些庞大的工具调用结果，并使用后台的大模型 (伴车 / Sidecar LLM) 对它们进行精简总结。接着，它会向主代理引擎“投影 (Project)”一段被重写过的对话历史——用一条简短密集的总结消息，来替换原本消耗大量 Token 的原始 `toolResult` 消息，保留结构。这是初级精简。如果内容越来越多，初级精简不够用，会直接用历史整体总结（根本不看 toolResult）取代思考和对话的全部消息，不保留结构，这个叫做高级精简。

## 核心架构与机制

### 1. 事件驱动的捕获与投影
该插件重度依赖 Pi Extension API (`pi.on`) 的事件钩子，而**绝不**永久性地篡改核心会话日志 (底层的日志始终是仅追加的)。
- **捕获**: 插件在每一回合结束时，将工具调用及其原始结果提取。
- **后台总结**: 如果 context_prune 被调用，将积压的批次发送给后台伴随大模型进行总结。插件自带重试和指数退避逻辑以应对并发限制。成功后，它会生成包含总结文本和对应 `toolCallId` 列表的数据。
- **历史投影**: 插件会拦截系统为 LLM 准备上下文请求的钩子。一旦发现某个消息的 `toolCallId` 已被总结过，就会将其移除。
- 所有 toolCall 会被压缩成一条摘要，放在最近一个 toolCallId 对应的位置，其他位置都删除。

### 2. 主动压缩防线 
- **使用率监控**: 在每个 `turn_end` 触发时，计算当前上下文窗口的使用率 (`tokens / contextWindow`)。
- **强制阻断阈值**: 如果当前上下文占用超过 `2/3` (66.66%)，插件将使用高级精简：要求伴车 LLM 对历史给出总结，并保持当前的任务。此时用猴子补丁过的上下文给伴车，让伴车浓缩成问题背景和当前进度即可，这个输出将替换主通道所有上下文。注意替换以后，主通道该停则还是停，该继续还是继续，不改变。

### 3. 大模型工具生态
- **(上下文修剪)**: 该工具向 LLM 暴露，无需参数。当 LLM 认为需要喘口气的时候调用它，它会触发后台进行总结，**并立刻返回执行权**，让主代理可以继续工作，绝不阻塞。

### 4. 用户可以选择伴随大模型 /pruner PROVIDER/MODEL

### 5.

- **零阻塞执行 (Zero-Block Execution)**: 坚守“伴随总结 (Sidecar)”的异步本质。主代理的主执行流永远不能被阻塞等待 `flushPending` 的大模型网络请求返回。
- **非破坏性历史投影 (Non-Destructive Projection)**: 必须且只能依赖 `context` 事件进行幻影替换。绝对不要尝试去直接删改底层的核心历史数组。
- **尊重提示词缓存 (Cache Awareness)**: 不断改写大模型前面的上下文会导致服务商侧的 Prompt Cache 彻底失效；把何时 flush 的决定权交给大模型。
